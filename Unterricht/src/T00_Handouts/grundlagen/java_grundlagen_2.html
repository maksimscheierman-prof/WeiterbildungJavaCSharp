1. Objekte

Was ist ein Objekt?
Ein Objekt ist eine Instanz einer Klasse — es kapselt Zustand (Attribute / Felder) und Verhalten (Methoden).

Analogie: Klasse = Bauplan, Objekt = tatsächliches Haus, das nach dem Plan gebaut wurde.

Beispiel (Java):

class Dog {
    String name;
    int age;

    void bark() {
        System.out.println(name + " bellt!");
    }
}

// Objekt erzeugen:
Dog d = new Dog();
d.name = "Bello";
d.age = 3;
d.bark(); // "Bello bellt!"

Wichtige Punkte:

  • Objekte haben Identität (z. B. Speicherreferenz).
  • Objekte können andere Objekte referenzieren.
  • Best Practices: Felder oft private + Zugriff über Getter/Setter — Kapselung.

2. Methoden — Signatur, Aufruf & Übergabe von Objekten

Methoden-Grundform

[modifiers] RückgabeTyp methodName([ParameterListe]) {
    // body
}

Beispiel:

public int add(int a, int b) {
    return a + b;
}

Übergabe von Objekten an Methoden — wichtiges Konzept

  • Java übergibt immer per Wert (pass-by-value).
  • Bei primitiven Typen (z. B. int) wird der Wert kopiert.
  • Bei Referenzen (Objekten) wird die Referenz-Kopie übergeben — resultierend:
    • Methoden können das Objekt verändern (á la obj.field = ...).
    • Methoden können die lokale Referenz neu zuweisen (obj = new ...), aber diese Neuzuweisung wirkt nicht auf die ursprüngliche Referenz des Aufrufenden.

Illustration (häufige Verwirrung):

class Test { int value; Test(int v) { value = v; } }

void modify(Test obj) {
    obj.value = 42;       // ändert das Originalobjekt
    obj = new Test(99);   // ändert nur die lokale Referenz 'obj'
}

Test t = new Test(5);
modify(t);
System.out.println(t.value); // -> 42

Methodenüberladung (overload) vs Überschreiben (override)

  • Overload: gleiche Methodennamen, unterschiedliche Parameter.
  • Override: Unterklasse implementiert Methode der Oberklasse / des Interfaces neu (@Override).

Übergabe von Collections / Arrays

  • Auch Collections werden per Referenzkopie übergeben — Änderungen am Objekt (z. B. list.add(...)) sind sichtbar.

3. Projektaufbau (gute Struktur)

Minimal empfehlenswerte Struktur (Maven/Gradle Konvention)

project-root/
 ├─ src/
 │  ├─ main/
 │  │  ├─ java/         // Anwendungscode
 │  │  └─ resources/    // configs, fxml, properties
 │  └─ test/
 │     └─ java/         // Unit-Tests
 ├─ pom.xml (Maven) oder build.gradle (Gradle)
 └─ README.md

Packages sinnvoll benutzen

  • Packages IMMER klein schreiben Gruppiere nach Funktion / Domäne: z. B.

    • com.example.app.model
    • com.example.app.controller
    • com.example.app.service
    • com.example.app.repository
    • com.example.app.interfaces

  • Vorteile: klare Verantwortungsbereiche, bessere Übersicht, vermeidet Namenskonflikte.

Tipps für gute Architektur

  • Layered architecture: UI / Controller → Service / Business logic → Repository / Data access
  • Single Responsibility Principle (SRP): Eine Klasse sollte nur eine Aufgabe haben.
  • Separation of Concerns (z. B. MVC für GUI).

  1. Klassen immer groß schreiben

  2. Package immer klein schreiben

  3. Methoden und Variablen camelCase

5. Warum float f = 2.0F; schreiben?

In Java sind Gleitkomma-Literale standardmäßig double.
float hat weniger Präzision und einen kleineren Wertebereich; um ein double-Literal in ein float zu packen, benötigst du eine Typangabe oder einen Cast.

Beispiele:

double d = 2.0;      // OK, 2.0 ist double
float f = 2.0;       // Fehler!!
float f1 = 2.0f;     // OK, 'f' suffix
float f2 = (float)2.0; // OK, mit Cast

Warum?
Ohne f oder Cast versucht Java, ein double → float implicit zu konvertieren, was einen möglichen Verlust bedeuten würde; Java verlangt deshalb eine explizite Angabe.

6. ++ / -- — post- und pre-increment / decrement

Vorzeichen (pre): ++i erhöht i sofort und liefert den neuen Wert.
Nachzeichen (post): i++ liefert erst den alten Wert, erhöht danach i.

Beispiele:

int i = 0;
int a = ++i; // i=1, a=1  (pre)
int j = 0;
int b = j++; // j=1, b=0  (post)

Usage-Pitfall:

int x = 5;
int y = x++ + ++x;
// Lese- und Schreibreihenfolge kann verwirren — vermeide solche Ausdrucke!

7. Dependency Injection (DI) — Pattern!

Kernidee:
Entkopple Klassen, indem du ihre Abhängigkeiten (z. B. ein Repository) von außen bereitstellst, statt dass die Klasse selbst sie erzeugt. Das macht Klassen testbar und flexibler.

Eine Klasse erstellt ihre Abhängigkeiten (z. B. Objekte) nicht selbst, sondern bekommt sie von außen übergeben (meist im Konstruktor).

Ohne Dependency Injection: 

class GameEngine {
    private Player player;

    public GameEngine() {
        this.player = new Player("Held", 100);
    }

    public void start() {
        player.attack();
    }
}

Problem:

  • GameEngine ist abhängig von der konkreten Klasse Player.

  • Wenn du später Mage oder Fighter nutzen willst, musst du den Code hier ändern.

  • Du kannst GameEngine nicht gut testen, weil du den Player nicht austauschen kannst.

Beispiel (klares, kleines Beispiel ohne Framework):

class GameEngine {
    private Player player;

    // Dependency Injection über den Konstruktor
    public GameEngine(Player player) {
        this.player = player;
    }

    public void start() {
        player.attack();
    }
}

In deiner main() entscheidest du dann, welcher Player verwendet wird:

Vorteile:

  • einfache Unit-Tests (Mocks können injiziert werden)
  • lose Kopplung -> Klassen sind Flexibler
  • bessere Modularität

8. Nested Classes (verschiedene Typen)

Sehr vorsichtig damit umgehen. Wird auch eher selten genutzt.

Java bietet mehrere Formen geschachtelter Klassen:

1. Static nested class

  • Ist wie eine normale Klasse, aber im Scope einer anderen.

  • Hat keinen impliziten Verweis auf die äußere Instanz.

    class Outer {
  static class StaticNested { }
}
Outer.StaticNested s = new Outer.StaticNested();

2. Inner class (non-static)

  • Jeder Inner-Instanz gehört eine Referenz zur äußeren Instanz.

    class Outer {
  class Inner { 
      void who() { System.out.println(Outer.this); }
  }
}
Outer o = new Outer();
Outer.Inner i = o.new Inner();

Nutzung: Wenn inner class eng an eine spezifische Außeninstanz gebunden ist (z. B. Iterator, GUI-Listener).

3. Local class

  • Definiert innerhalb einer Methode — haben Zugriff auf final/effectively final lokale Variablen.
  • Werden selten verwendet. Normalerweise, wenn man komplexe Datenstrukturen immer innerhalb einer Methode haben will und sie nicht von außen sichtbar sein sollen.

4. Anonymous class

  • Kurzlebige Klasse für einmalige Implementierung (oft für Listener).

    Runnable r = new Runnable() {
  public void run() { System.out.println("Run"); }
};

9. Safety / defensive programming & Optimierungen

Defensive Programming (Sicherheits- & Stabilitätsmaßnahmen)

  • Input-Validierung: Prüfe Benutzereingaben immer.

  • Null-Safety: Prüfe null (oder nutze Optional).

  • Immutability: Verwende final für Felder, gebe keine internen Collections direkt zurück (defensive copy).

  • Schütze interne Daten:

    public List<Item> getItems() {
    return new ArrayList<>(this.items); // Defensive copy
}

  • Ressourcen sauber schließen: try-with-resources statt manueller close():

    try (BufferedReader r = new BufferedReader(...)) {   //Try-with-resources. Resourcen werden nach dem Try-Block geschlossen
    // read
}

  • Fehlerbehandlung: sinnvolles Logging + gezielte Exceptions (keine leeren catch {} Blöcke).

Performance-Optimierungen (nicht vorzeitig!)

  • StringBuilder bei vielen String-Konkatenationen.
  • Vermeide unnötiges Boxing/Unboxing (z. B. Integer vs int).
  • Datenstrukturauswahl: ArrayList vs LinkedList vs HashMap je nach Use-Case.

Sicherheitsoptimierungen (Security)

  • Eingabesanitierung (SQL Injection, XSS).
  • Least Privilege: Programme laufen mit minimal notwendigen Rechten.
  • Sensiblen Daten schützen (keine Passwörter in Logs)

10. Polymorphismus

Der Begriff „Polymorphismus“ bezeichnet in der objektorientiertenProgrammierung die Fähigkeit von Objekten, sich je nach ihrem konkreten Typ unterschiedlich zu verhalten, obwohl sie über eine gemeinsame Schnittstelle (z.B. eine Oberklasse oder ein Interface) angesprochen werden.

Compile-Time-Polymorphismus (Method Overloading):

  • Mehrere Methoden im selben Gültigkeitsbereich besitzen denselben Namen, aber unterschiedliche Parameterlisten (Anzahl, Typ oder Reihenfolge).
  • Die Entscheidung, welche Methode aufgerufen wird, erfolgt zur Compile-Zeit.
  • Beispiel:
class Rechner {
    static int addiere(int a, int b) {
        return a + b;
    }

    static double addiere(double a, double b) {
        return a + b;
    }

    public static void main(String[] args) {
        addiere(1,1);
        addiere(2.0, 2.0);
    }
}

Runtime-Polymorphismus (MethodOverriding):

  • Eine Methode der Oberklasse wird in einer Unterklasse überschrieben (gleicheSignatur).
  • Der tatsächliche Methodenaufruf wird zur Laufzeit bestimmt (dynamischeBindung).
  • Beispiel:

class Tier {
    void makeSound() {
        System.out.println("Ein Tiermacht ein Geräusch");
    }
}

class Hund extends Tier {
    void makeSound() {
        System.out.println("Der Hundbellt");
    }
}

class Main {
    public static void main(String[] args) {
        Tier t = new Hund();// Polymorphismus
        t.makeSound(); // Ausgabe: "Der Hund bellt"
    }
}
  • Polymorphismus funktioniert über Vererbung oder Interfaces.
  • Nur Methoden können polymorph aufgerufen werden, nicht Attribute.
  • Welche Methode tatsächlich ausgeführt wird, hängt vom Typ des Objekts ab (nicht vom Typ der Referenzvariablen).
  • Polymorphismus ist ein Grundpfeiler der objektorientierten Programmierung, neben Kapselung und Vererbung.

InstanceOf

Das Schlüsselwort instanceof prüft, ob ein Objekt eine Instanz einer bestimmten Klasse oder deren Unterklasse ist.
Es gibt einen boolean (true/false) zurück.

for (Medium m : medienListe) {
    if (m instanceof Buch) {
        Buch b = (Buch) m; // Typumwandlung
        System.out.println("Autor: " + b.getTitle());
    } else if (m instanceof Zeitschrift) {
        Zeitschrift z = (Zeitschrift) m;
        System.out.println("Ausgabe: " + z.getTitle());
    }
}

Warum instanceof wichtig ist

  • Wird genutzt, wenn du unterschiedliche Objekttypen in einer gemeinsamen Sammlung hast (z. B. ArrayList<Medium>).

  • Hilft, den konkreten Typ zur Laufzeit zu erkennen.

  • Wird oft zusammen mit Casting verwendet.

11. Das Schlüsselwort "throws" im Methodenkopf

In Java dient das Schlüsselwort throws dazu, anzugeben, dass eine Methode bestimmte Ausnahmen (Exceptions) an den Aufrufer weitergeben kann, anstatt sie selbst zu behandeln.

Beispiel:

import java.io .*;

public class DateiLeser {
    public void ladeDatei(String pfad) throws IOException {
        FileReader reader = newFileReader(pfad);
        System.out.println("Dateierfolgreich geladen!");
    }
}

  • Mit „throws IOException“ teilt die Methode mit, dass beim Ausführen eine IOException auftreten kann.
  • Der Aufrufer der Methode muss die Ausnahme behandeln (try-catch) oder selbst weiterwerfen.
  • Checked Exceptions (z. B. IOException, SQLException) müssen entweder behandelt oder deklariert werden.
  • Unchecked Exceptions (RuntimeExceptions wie NullPointerException) müssen nicht angegeben werden.

Beispiel mit try-catch:

 public static void main (String[]args){
            DateiLeser leser = new DateiLeser();
            try {
                leser.ladeDatei("text.txt");
            } catch (IOException e) {
                System.out.println("Fehlerbeim Laden der Datei: " + e.getMessage());
            }
        }
